## Architecture

```
Browser
  │
  ▼
frontend container (Nginx) — port 8082
  │  serves static HTML/CSS/JS
  │  proxies /api/* → backend:5000 (internal Docker network only)
  │
  ▼
backend container (Flask) — port 5000 (internal only, NOT exposed)
  │  applies business logic
  │  inserts record into MySQL
  │
  ▼
mysql container — port 3306 (internal only, NOT exposed)
  │  persists data in Docker named volume: mysql_data
```

Key design decisions:
- Backend port `5000` is **not exposed** to the host — only reachable via Nginx proxy inside Docker network
- Frontend uses **relative URL** `/api/customers` in JS — works identically on localhost and EC2
- Nginx config (`frontend/nginx.conf`) handles the proxy from frontend to backend

## Business Logic

| Rule | Where | Example |
|---|---|---|
| `some_number × 2` | `frontend/script.js` — before POST | User enters 10 → 20 sent to backend |
| `age + 1` | `backend/app.py` — after receiving | User enters 25 → 26 stored in DB |
| `submitted_at` | `backend/app.py` — at insert time | Auto UTC timestamp, user never sets this |

## Local Development Setup

## AWS EC2 Setup

### Step 1 — Launch EC2 Instance

1. Go to AWS Console → EC2 → Launch Instance
2. Choose **Amazon Linux 2 AMI**
3. Select instance type (t2.micro for testing, t2.small or above for production)
4. Configure storage — set disk size based on expected data (e.g. 100GB+)
5. Configure Security Group — add inbound rules:

| Type | Port | Source | Purpose |
|---|---|---|---|
| SSH | 22 | Your IP | SSH access |
| Custom TCP | 8082 | 0.0.0.0/0 | Web application |

6. Create or select a key pair (.pem file) — save it safely
7. Launch the instance

### Step 2 — SSH into EC2

```bash
chmod 400 your-key.pem
ssh -i your-key.pem ec2-user@<EC2-PUBLIC-IP>
```

## Docker & Docker Compose Installation

Run these commands after SSH into EC2:

```bash
# Update system
sudo yum update -y

# Install Docker
sudo yum install docker -y

# Start Docker service
sudo service docker start

# Enable Docker to start on reboot
sudo systemctl enable docker

# Add ec2-user to docker group (so you can run docker without sudo)
sudo usermod -aG docker ec2-user

# IMPORTANT: Log out and back in for group change to take effect
exit
# SSH back in again
ssh -i your-key.pem ec2-user@<EC2-PUBLIC-IP>

# Verify Docker works
docker --version
docker ps
```

### Install Docker Compose v1

```bash
sudo curl -L "https://github.com/docker/compose/releases/latest/download/docker-compose-$(uname -s)-$(uname -m)" \
  -o /usr/local/bin/docker-compose

sudo chmod +x /usr/local/bin/docker-compose

# Verify
docker-compose --version
```

> **Note:** This project uses `docker-compose` (v1 syntax with hyphen). If your server has Docker Compose v2 (`docker compose` with space), update the CD workflow commands accordingly.

## CI/CD Pipeline

### Overview

```
Developer pushes code
        │
        ▼
  Trigger CI manually
  (GitHub Actions tab)
        │
        ▼
┌─────────────────────────────┐
│  CI Workflow (ci.yml)       │
│  1. Run 18 pytest tests     │
│  2. Build backend image     │
│  3. Build frontend image    │
│  4. Push both to Docker Hub │
│     - :latest               │
│     - :<git-sha>            │
└─────────────────────────────┘
        │ on success
        ▼
┌─────────────────────────────────────────┐
│  CD Workflow (cd.yml)                   │
│  1. SSH into EC2                        │
│  2. Clone repo (first time) or git pull │
│  3. Auto-generate .env from secrets     │
│  4. docker pull :latest images          │
│  5. docker-compose down                 │
│  6. docker-compose up -d               │
│  7. Health check (5 retries)            │
│  8. Print web URL in logs               │
└─────────────────────────────────────────┘
```

### CI Trigger

Go to GitHub → your repo → **Actions** tab → **CI — Test, Build & Push Docker Images** → **Run workflow** → **Run workflow**

### Image Tags

Every CI run produces two Docker Hub tags:
- `pawarakash2511/customer-api:<git-sha>` — unique, immutable, traceable per commit
- `pawarakash2511/customer-api:latest` — always points to the newest build (what CD pulls)

### .env Auto-Generation

CD automatically writes the `.env` file on EC2 from GitHub Secrets on every deploy. You never need to manually create it:

```bash
# Generated automatically by CD workflow:
DB_HOST=mysql
DB_USER=root
DB_PASSWORD=<from DB_PASSWORD secret>
DB_NAME=customerdb
MYSQL_ROOT_PASSWORD=<from DB_PASSWORD secret>
```

Go to: GitHub repo → **Settings** → **Secrets and variables** → **Actions** → **New repository secret**

|---|---|---|
| `DOCKER_USERNAME` | `pawarakash2511` | Docker Hub login |

| `EC2_HOST` | EC2 public IP (e.g. `13.234.x.x`) | SSH target |
| `EC2_USER` | `ec2-user` | SSH username |

## Admin Panel

A password-protected dashboard for viewing all submitted customer records.

### How to access

1. Open the main app — click **"Admin Panel"** button in the top-right corner of the form
2. Or navigate directly to `/admin.html`

| Environment | URL |
|---|---|
| Local | `http://localhost:8082/admin.html` |
| EC2 | `http://<EC2-PUBLIC-IP>:8082/admin.html` |

### Login

Enter admin credentials on the login screen. On success, a JWT token (8-hour expiry) is stored in `sessionStorage` and the dashboard loads automatically.

> Credentials are set via `ADMIN_USERNAME` and `ADMIN_PASSWORD` environment variables. Default values are configured in `backend/app.py`.

### Dashboard features

| Feature | Description |
|---|---|
| Customer records table | Shows all rows from the `customers` table, newest first |
| Columns | #, Customer ID, Name, Gender, Age (stored), Some Number (stored), Submitted At (UTC) |
| Refresh | Reloads data from the database without logging out |
| Logout | Clears the JWT token from `sessionStorage` and returns to login screen |
| Session expiry | If the 8-hour token expires, any data request automatically redirects to login |

### Security

- JWT signed with `HS256` — token is verified on every API call
- Token stored in `sessionStorage` — cleared automatically when the browser tab closes
- All data in the table is HTML-escaped to prevent XSS
- Backend credentials overridable via env vars: `ADMIN_USERNAME`, `ADMIN_PASSWORD`, `JWT_SECRET`

### New backend endpoints

| Method | Endpoint | Auth | Description |
|---|---|---|---|
| `POST` | `/api/admin/login` | None | Validates credentials, returns JWT token |
| `GET` | `/api/admin/customers` | Bearer JWT | Returns all customer records as JSON |

## API Reference

### POST /api/customers

Stores a customer record.

**Request:**
```json
{
  "customer_id": "CUST001",
  "customer_name": "John Doe",
  "gender": "Male",
  "age": 25,
  "some_number": 20
}
```

**Success Response (201):**
```json
{
  "status": "success",
  "message": "Customer data saved successfully"
}
```

**Error Response (400):**
```json
{
  "status": "error",
  "message": "Missing fields: customer_id"
}
```

**Error Response (500):**
```json
{
  "status": "error",
  "message": "DB connection failed"
}
```

### Validation Rules
- All 5 fields are required
- Empty strings are rejected
- `age` must be a valid number (min 0)
- `gender` must be one of: Male, Female, Other

---

### POST /api/admin/login

Authenticates the admin and returns a JWT token.

**Request:**
```json
{
  "username": "AkashPawar",
  "password": "Akash34567"
}
```

**Success Response (200):**
```json
{
  "status": "success",
  "token": "<JWT token>"
}
```

**Error Response (401):**
```json
{
  "status": "error",
  "message": "Invalid username or password"
}
```

---

### GET /api/admin/customers

Returns all customer records. Requires a valid JWT token in the `Authorization` header.

**Request header:**
```
Authorization: Bearer <token>
```

**Success Response (200):**
```json
{
  "status": "success",
  "count": 3,
  "data": [
    {
      "id": 3,
      "customer_id": "CUST003",
      "customer_name": "Jane Smith",
      "gender": "Female",
      "age": 31,
      "some_number": 20,
      "submitted_at": "2026-06-04 10:23:45"
    }
  ]
}
```

**Error Response (401):**
```json
{
  "status": "error",
  "message": "Token required"
}
```

## Troubleshooting

### Issue 1: `docker: 'compose' is not a docker command`

**Cause:** Server has Docker Compose v1 installed (standalone binary) not v2 (plugin).

**Fix:** Use `docker-compose` (with hyphen) instead of `docker compose` (with space).

```bash
# Wrong (v2 syntax)
docker compose up -d

# Correct (v1 syntax)
docker-compose up -d
```

The CD workflow in this project already uses v1 syntax (`docker-compose`).

---

### Issue 2: `Bind for 0.0.0.0:80 failed: port is already allocated`

**Cause:** Another container is already using port 80 on the EC2 server.

**Fix:** Change the frontend port in `docker-compose.yml` to a free port (this project uses `8082`):

```yaml
frontend:
  ports:
    - "8082:80"   # host:container
```

Then open port `8082` in AWS Security Group (EC2 → Security Groups → Inbound Rules → Add Rule → Custom TCP → 8082).

---

### Issue 3: Form submits but API call fails (CORS / localhost error)

**Cause:** `script.js` was using `http://localhost:5000` as the API URL. In production, `localhost` points to the user's browser machine, not the server.

**Fix:** Use a relative URL and configure Nginx to proxy API calls:

```js
// Wrong
const API_URL = "http://localhost:5000/api/customers";

// Correct
const API_URL = "/api/customers";
```

`frontend/nginx.conf` proxies `/api/` to the backend container internally:

```nginx
location /api/ {
    proxy_pass http://backend:5000/api/;
}
```

---

### Issue 4: Health check fails after deployment

**Check container status:**
```bash
docker ps
docker-compose ps
```

**Check logs:**
```bash
docker-compose logs backend
docker-compose logs frontend
docker-compose logs mysql
```

**Restart specific service:**
```bash
docker-compose restart backend
```

**Full restart:**
```bash
docker-compose down
docker-compose up -d
```

---

### Issue 5: MySQL container keeps restarting

**Cause:** Usually a wrong password in `.env` or volume corruption.

**Check logs:**
```bash
docker-compose logs mysql
```

**Reset MySQL volume (wipes all data — use carefully):**
```bash
docker-compose down -v
docker-compose up -d
```

---

### Issue 6: CD pipeline fails — `Permission denied` on SSH

**Cause:** `EC2_SSH_KEY` secret is missing the full key content or has extra spaces.

**Fix:**
- Open `.pem` file in a plain text editor
- Copy the entire content including header and footer lines
- Paste into the GitHub Secret with no extra spaces or newlines

---

### Issue 7: Images not updating after push

**Cause:** Docker caches old layers.

**Fix — force pull on EC2:**
```bash
docker-compose down
docker pull pawarakash2511/customer-api:latest
docker pull pawarakash2511/customer-ui:latest
docker-compose up -d
```

## Responsive Design

The frontend works on all devices. Key techniques applied:

| Technique | File | Purpose |
|---|---|---|
| `viewport-fit=cover` | `index.html` | Extends into iPhone notch/Dynamic Island area |
| `env(safe-area-inset-*)` | `style.css` | Adds padding so content isn't hidden by notch |
| `font-size: 16px` on inputs | `style.css` | Prevents iOS auto-zoom on input focus |
| `min-height: 44px` | `style.css` | Apple/Android minimum touch target size |
| `100dvh` | `style.css` | Fixes iOS Safari address bar viewport issue |
| `inputmode="numeric"` | `index.html` | Shows numeric keypad on mobile for number fields |
| `-webkit-text-size-adjust: 100%` | `style.css` | Prevents font inflation on orientation change |
| `touch-action: manipulation` | `style.css` | Removes 300ms tap delay on mobile |
| Breakpoints: 768px, 520px, 375px | `style.css` | Tablet, phone, small phone (iPhone SE) |
